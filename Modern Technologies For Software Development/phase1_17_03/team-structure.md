# 3. Опис ролей та відповідальностей


## Яцків Віталій — Frontend Engineer / QA

Frontend Engineer відповідає за користувацький інтерфейс застосунку.

### Основні обов’язки

- реалізація інтерфейсу користувача
- інтеграція з backend API
- створення сторінок та компонентів
- забезпечення адаптивного дизайну
- покращення користувацького досвіду (UX)

### Зона прийняття рішень

- структура frontend застосунку
- дизайн компонентів
- організація клієнтської логіки

---

## Процьків Назарій — Tech Lead / Backend Engineer

Tech Lead відповідає за технічну архітектуру системи та координує процес розробки.

### Основні обов’язки

- визначення архітектури системи
- вибір технологій та інструментів
- контроль якості коду
- проведення code review
- допомога іншим розробникам у вирішенні технічних проблем
- координація backend частини системи
- прийняття технічних рішень
- розробка API
- реалізація бізнес-логіки
- робота з базою даних
- інтеграція із зовнішніми сервісами (наприклад курси валют)
- оптимізація продуктивності сервера

### Зона прийняття рішень

- архітектура системи
- стандарти кодування
- структура проєкту
- реалізація серверної логіки
- структура API

---

## Карпа Юрій — Backend Engineer

Backend Engineer відповідає за реалізацію серверної логіки застосунку.

### Основні обов’язки

- розробка API
- реалізація бізнес-логіки
- робота з базою даних
- інтеграція із зовнішніми сервісами
- оптимізація продуктивності сервера

### Зона прийняття рішень

- реалізація серверної логіки
- структура API
- оптимізація запитів до бази даних

---


## Зеленько Ірина — Product Owner / Designer 

Product Owner відповідає за управління вимогами та перевірку якості продукту.

### Основні обов’язки

- формування вимог до системи
- створення та підтримка backlog
- написання user stories
- тестування функціоналу
- перевірка відповідності реалізації вимогам
- комунікація між членами команди

### Зона прийняття рішень

- пріоритети функціональності
- формування вимог
- прийняття функціоналу після тестування

---

# 4. Формат взаємодії в команді

Команда використовує **GitHub** як основний інструмент для управління проєктом.

### Формат взаємодії

- код зберігається у GitHub репозиторії
- кожна нова функція реалізується у окремій гілці
- зміни інтегруються через Pull Request
- перед злиттям коду проводиться code review
- задачі фіксуються у вигляді GitHub Issues

---

# 5. Процес прийняття рішень

У команді використовується **колективний підхід до прийняття рішень**.

### Принципи

- технічні рішення приймає **Tech Lead** після обговорення з командою
- продуктні рішення приймає **Product Owner**
- складні рішення приймаються шляхом **обговорення та голосування**
- у разі суперечностей остаточне рішення приймається спільно командою

---

# 6. Командні принципи роботи

Команда дотримується наступних принципів:

- відкритість до обговорення
- відповідальність за власні задачі
- регулярна комунікація між учасниками
- дотримання дедлайнів
- підтримка один одного у процесі розробки

---

# 7. Інструменти команди

Команда використовує наступні інструменти:

- **GitHub** — система контролю версій та управління кодом
- **Figma** — створення дизайну інтерфейсу
- **Discord** — комунікація між учасниками
- Інші сервіси — для створення UML та інших діаграм
